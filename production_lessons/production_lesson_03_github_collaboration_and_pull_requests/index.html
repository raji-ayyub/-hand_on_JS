GitHub collaboration and pull requests

This lesson connects your local Git workflow to team collaboration: remotes, pushing, review, and merging.

Back to production track

Git vs GitHub

  • Git tracks history in your local repository.
  • GitHub hosts repositories and supports collaboration, review, and sharing.
  • You can use Git without GitHub, but teams often use both together.

Team rule of thumb

Do your work on a feature branch, open a pull request, and merge after review instead of committing directly to main.

Common collaboration flow

git clone <repository-url>
git switch -c feature/search-filter
# make changes
git add .
git commit -m "Add company filter to user directory"
git push -u origin feature/search-filter

After pushing, open a pull request on GitHub and describe the change clearly.

Good pull request structure

  • Short, specific title
  • Summary of the problem and the change
  • Testing notes
  • Screenshots if the UI changed
  • Known limitations if anything is still pending

Useful remote commands

git remote -v
git fetch origin
git pull origin main
git push origin branch-name

Example pull request description

Title:
Add company filter to capstone user directory

Summary:
- added a select input for company filtering
- updated app state to store selected company
- filtered rendered users by search term and company

Testing:
- searched by name
- switched between company options
- verified empty-state message still appears correctly

Practice prompts

  • Write three branch names for three imaginary frontend tasks.
  • Write a pull request description for one change in this repo.
  • Pick one small change and describe how you would test it before asking for review.
  • Review one of your own commits and write two helpful review comments.
